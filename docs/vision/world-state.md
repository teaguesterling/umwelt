# umwelt: The World State Layer

*How the world the view attaches policy to gets defined, discovered, projected, and materialized. The DOM to umwelt's CSS — the thing selectors match against, the thing compilers walk, the thing auditors snapshot.*

---

## 1. The three-layer architecture

umwelt's design mirrors the web platform's foundational separation:

| Layer | Web analogy | umwelt | Format | Purpose |
|---|---|---|---|---|
| **Vocabulary** | HTML DTD | `@entity`, `@attribute`, `@property` at-rules in `.umw` files | CSS at-rules | What kinds of things can exist and what properties they can carry |
| **World State** | The DOM | `.world.yml` files | YAML | What things actually exist right now — the concrete entities selectors match against |
| **Policy** | CSS stylesheets | `.umw` view files | CSS | What rules apply to matched entities |

The separation is load-bearing. Each layer has a different rate of change, a different author, and a different reason to exist:

- **Vocabulary** changes when the system's conceptual model evolves (new entity types, new property kinds). Authored by tool developers. Changes rarely.
- **World state** changes when the environment changes (new files, different tools available, different principal). Authored by operators, generated by discovery, or composed from includes. Changes per-session or per-delegation.
- **Policy** changes when the rules change (tighter sandbox, new mode, ratchet turn). Authored by principals or generated by the ratchet. Changes per-task or per-observation.

Mixing them produces the same pathology as inline styles in HTML: the policy is entangled with the world, and you can't reuse either independently.

## 2. Vocabulary: CSS at-rules

Vocabulary declares what *kinds* of things can exist. It's the type system — the DTD that makes the DOM well-formed.

### 2.1 Format

Vocabulary is declared in `.umw` files using CSS at-rules. This is precedented by CSS Houdini's `@property` rule, which lets stylesheets declare custom property types before using them.

```css
/* Declare an entity type */
@entity file {
  taxon: world;
  parent: dir;
  attributes: name, path, language, size, mtime;
}

@entity tool {
  taxon: capability;
  attributes: name, description;
}

@entity mode {
  taxon: state;
  attributes: name;
}

/* Declare attribute schemas */
@attribute path {
  type: string;
  comparison: glob;    /* enables file[path~="src/**"] */
}

@attribute size {
  type: integer;
  comparison: numeric; /* enables file[size>1024] */
}

/* Declare property types with comparison semantics */
@property editable {
  type: boolean;
  comparison: exact;   /* last-wins by specificity */
  default: false;
}

@property limit {
  type: quantity;
  comparison: cap;     /* <= takes minimum */
}

@property allowed-commands {
  type: set;
  comparison: pattern; /* union of matching patterns */
}
```

### 2.2 Design rationale

CSS at-rules for vocabulary, not a separate format, because:

1. **One parser.** The tinycss2 parser already handles at-rules. No second parser, no second grammar, no format boundary.
2. **Co-location is optional.** Vocabulary at-rules can live in a dedicated `vocabulary.umw` file or inline in the same file as policy. The parser doesn't care.
3. **Forward compatibility.** Unknown at-rules are preserved with a warning — the same forward-compat mechanism views already use.
4. **Houdini precedent.** `@property` in CSS Houdini does exactly this: declares type information in the stylesheet's own syntax so the cascade can reason about it.

### 2.3 Relationship to plugin registration

Today, vocabulary is registered in Python via `register_entity()`, `register_taxon()`, etc. The at-rule declarations are the *declarative surface* for the same registration — they compile to the same `EntitySchema` and `AttrSchema` objects. Python registration remains available for consumers that need programmatic control.

## 3. World State: YAML

World state declares what *actually exists* — the concrete entities that selectors match against. It's the DOM that CSS styles.

### 3.1 Why YAML

The world file is deliberately not CSS. Three reasons:

1. **No conditionals.** YAML is a data format, not a rule language. You can't write `if mode == "implement" then add tool X` in YAML. Conditionals belong in policy (CSS), not in world state. The format enforces the separation.
2. **Composition.** YAML has established patterns for file inclusion, fragment references, and anchors. World definitions need to compose (base world + project-specific overlay + delegation-specific restrictions). CSS has no native inclusion mechanism.
3. **Familiarity.** Docker Compose, Kubernetes, GitHub Actions, Ansible — operators already think in YAML for environment declaration. The mental model transfers.

### 3.2 File format

World files use the extension `.world.yml`. A complete example:

```yaml
# delegate-sandbox.world.yml
#
# The world a code-editing delegate operates inside.

# --- Explicit entities ---
# Named, hand-declared. The "docker run" arguments.
entities:
  - type: principal
    id: developer
    classes: [human, owner]
    attributes:
      email: "dev@example.com"

  - type: mode
    id: implement
    classes: [edit, test]

  - type: tool
    id: Read
    attributes:
      description: "Read file contents"

  - type: tool
    id: Edit
    attributes:
      description: "Edit file contents"

  - type: tool
    id: Bash
    classes: [dangerous]
    attributes:
      description: "Execute shell commands"

# --- Discovery recipes ---
# Implicit enumeration. The "COPY" / "ADD" directives.
# Each recipe names a matcher and its configuration.
discover:
  - matcher: filesystem
    root: "src/"
    include: ["**/*.py", "**/*.ts"]
    exclude: ["**/node_modules/**", "**/__pycache__/**"]

  - matcher: filesystem
    root: "tests/"
    include: ["**/*.py"]

  - matcher: git-tracked
    root: "."
    respect-gitignore: true

# --- Projections ---
# External data mounted into the world without full enumeration.
# Like Docker bind mounts: the world references them, but doesn't
# list every file inside.
projections:
  - type: dir
    id: node_modules
    attributes:
      path: "node_modules/"
    # Entities inside are not enumerated — policy can still
    # match on the projection itself (e.g., dir#node_modules { editable: false })

  - type: network
    id: pypi
    attributes:
      host: "pypi.org"
      protocol: https

# --- Overrides ---
# World-level attribute overrides. Selector-as-key.
# These modify entity attributes after discovery, before policy.
# Use case: the discovered filesystem says auth.py is 200 lines,
# but you want to mark it as security-critical.
overrides:
  "file[path='src/core/auth.py']":
    classes: [security-critical]
    attributes:
      review-required: true

  "tool#Bash":
    classes: [restricted]

# --- Fixed constraints ---
# Hard boundaries that policy CANNOT override.
# Selector-as-key, same as overrides.
# These are world-level facts: the entity doesn't exist in this
# world, or it has an immutable attribute. Policy can add further
# restrictions but cannot relax fixed constraints.
fixed:
  "tool#Bash":
    available: false

  "network":
    deny: "*"

  "file[path~='**/.env']":
    visible: false

# --- Composition ---
# Include other world files. Later files override earlier ones.
# Supports fragment references for partial inclusion.
include:
  - "base-python.world.yml"
  - "team-tools.world.yml#tools"
  - "project-specific.world.yml"
```

### 3.3 Shorthand syntax

For common entity types, top-level shorthand keys desugar to `entities:` entries:

```yaml
# Shorthand form
tools: [Read, Edit, Grep]
modes: [implement]
principal: Teague
inferencer: claude-sonnet-4-6
resources:
  memory: 512MB
  wall-time: 5m
```

This is equivalent to:

```yaml
entities:
  - type: tool
    id: Read
  - type: tool
    id: Edit
  - type: tool
    id: Grep
  - type: mode
    id: implement
  - type: principal
    id: Teague
  - type: inferencer
    id: claude-sonnet-4-6
  - type: resource
    id: memory
    attributes: { limit: 512MB }
  - type: resource
    id: wall-time
    attributes: { limit: 5m }
```

Shorthands and `entities:` can be mixed in the same file. Shorthands are expanded before `entities:` is processed, so explicit entries can override shorthand values.

The shorthand keys are determined by the registered vocabulary — each taxon can declare which of its entity types support shorthand syntax. The sandbox vocabulary registers shorthands for `tool`, `mode`, `principal`, `inferencer`, and `resource`.

### 3.4 Section semantics

#### `entities:` — explicit declaration

Named entities with explicit types, IDs, classes, and attributes. These are always present in the world regardless of what discovery finds. Analogous to `docker run` arguments or Terraform resources — you declare what exists.

Each entity is a map with:
- `type` (required): entity type name from the vocabulary → maps to `entities.type_name` in SQL
- `id` (required): unique identifier within type → maps to `entities.entity_id` in SQL
- `classes` (optional): list of class names for class selectors (`.implement`, `.dangerous`) → JSON array in `entities.classes`
- `attributes` (optional): map of attribute values for attribute selectors (`[path^="src/"]`) → JSON object in `entities.attributes`
- `parent` (optional): ID of parent entity for hierarchy → `entities.parent_id` FK

#### `discover:` — implicit enumeration

Recipes that enumerate entities from external sources. Each recipe names a registered matcher and passes configuration to it. Matchers are plugins — the filesystem matcher scans directories, the git-tracked matcher reads `.gitignore`, future matchers could scan package registries, API schemas, or running processes.

Discovery runs at world-materialization time. The results are entity rows, identical in shape to explicit entities. Explicit entities and discovered entities coexist; if both declare the same `(type, id)` pair, the explicit declaration wins.

Matchers registered via umwelt's plugin system participate automatically. No hardcoded matcher list.

**Matcher protocol extension.** Today's `MatcherProtocol` has `match_type(type_name) -> list[Entity]` with no configuration. Discovery recipes require extending the protocol to accept configuration (root paths, include/exclude patterns, depth limits). The existing matchers continue to work unchanged — discovery configuration is an optional capability layered on top.

#### `projections:` — mounted references

External data referenced by the world without enumerating its contents. The projection itself is an entity (matchable by selectors), but its children are not listed. Policy can attach rules to the projection entity; enforcement tools decide how to handle the un-enumerated interior.

Projections are the world-state equivalent of Docker bind mounts: you declare that a volume exists and where it's mounted, but you don't list every file inside. This keeps world files tractable for large external trees (node_modules, system libraries, network zones).

A future version may support lazy enumeration of projection contents on demand.

#### `overrides:` — post-discovery attribute modification

Selector-as-key maps that modify entity attributes after discovery but before policy evaluation. The selector identifies which entities to modify; the value is a map of attributes and classes to set.

Overrides are world-level facts, not policy. They say "this entity *is* security-critical" not "this entity *should be treated as* security-critical." The distinction matters for auditing: overrides appear in the world snapshot, policy appears in the cascade.

#### `fixed:` — immutable constraints

Selector-as-key maps that declare hard boundaries policy cannot override. Fixed constraints are world-level truths — they describe what *doesn't exist* or what *cannot change* in this world.

```yaml
fixed:
  "tool#Bash":
    available: false        # Bash does not exist in this world
  "network":
    deny: "*"               # No network access, period
  "file[path~='**/.env']":
    visible: false           # .env files are not visible
```

The enforcement semantics: if policy says `tool#Bash { available: true; }` and `fixed` says `"tool#Bash": { available: false }`, the fixed constraint wins. Fixed constraints are not part of the cascade — they are applied after cascade resolution as a final clamp.

This is analogous to hardware constraints: you can write CSS that says `width: 5000px` but the physical screen is 1920px wide. Fixed constraints are the physical screen.

#### `include:` — composition

A list of other `.world.yml` files to include. Included files are processed in order; later declarations override earlier ones for the same `(type, id)` pair. Fragment references (`file.yml#section`) include only the named section.

Include enables the layering pattern:

```yaml
# base-python.world.yml — shared across all Python projects
# project.world.yml — project-specific entities and discovery
# delegation.world.yml — per-delegation restrictions

include:
  - "base-python.world.yml"
  - "project.world.yml"
```

The base world defines common tools, modes, and discovery patterns. The project world adds project-specific entities. The delegation world narrows for a specific task. Each layer is independently authored and versioned.

**Entity removal.** To *remove* an entity from an included base (rather than override it), use the `exclude:` key alongside `include:`:

```yaml
include:
  - "base-tools.world.yml"

exclude:
  - "tool#Bash"          # remove Bash from the included tools
  - "tool#Write"         # remove Write too
```

Exclusions are processed after all includes are merged. The excluded entities are removed from the world entirely — they cannot be matched by policy. This is the "narrowing" pattern for delegation: include a broad base, then exclude what the delegate shouldn't have.

### 3.5 The container analogy

A world file is a container spec for an agent's environment:

| Container concept | World file concept |
|---|---|
| Base image | `include:` of a base world |
| `COPY` / `ADD` | `discover:` recipes |
| `docker run` args | `entities:` explicit declarations |
| Bind mounts | `projections:` |
| `ENV` / build args | `vars:` (reserved, future) |
| Read-only mounts | `fixed:` constraints |
| Multi-stage builds | Multiple `include:` layers |

The analogy is not cosmetic. Both systems solve the same problem: declaring an isolated environment with specific contents, capabilities, and constraints, in a composable way that separates the environment definition from what runs inside it.

## 4. Materialization

A world file is a recipe. **Materialization** is what you get when you execute the recipe — a concrete snapshot of every entity in the world, with provenance.

### 4.1 Why materialize

Three purposes:

1. **Audit.** "What could the delegate see?" is a security question. The materialized world is the answer — a complete inventory, not a set of recipes to re-run.
2. **Reproducibility.** Two runs of the same world file may produce different results if the filesystem changed. A materialized snapshot pins the world to a specific point in time.
3. **SQL compilation.** The SQL compiler needs concrete entity rows to INSERT. Materialization produces them.

### 4.2 Detail levels

Not every consumer needs every file listed. Materialization supports three detail levels:

| Level | What's included | Use case |
|---|---|---|
| `summary` | Entity counts per type, projection boundaries, fixed constraints | Quick audit, dashboard |
| `outline` | Entity types + IDs + classes, no attributes | Session recovery, diff |
| `full` | Every entity with all attributes | SQL compilation, deep audit |

```bash
# Materialize at different detail levels
umwelt materialize delegate.world.yml                    # default: full
umwelt materialize delegate.world.yml --level summary    # counts only
umwelt materialize delegate.world.yml --level outline    # types + IDs
umwelt materialize delegate.world.yml -o snapshot.yml    # save to file
```

### 4.3 Materialized format

The output is YAML — the same shape as the `entities:` section, but with every entity expanded and annotated with provenance:

```yaml
# Materialized world state — 2026-04-18T14:32:00Z
meta:
  source: delegate-sandbox.world.yml
  materialized_at: "2026-04-18T14:32:00Z"
  detail_level: full
  entity_count: 847
  includes:
    - base-python.world.yml
    - project.world.yml

entities:
  - type: file
    id: "src/core/auth.py"
    classes: [security-critical]
    attributes:
      path: "src/core/auth.py"
      language: python
      size: 4821
      mtime: "2026-04-17T09:15:00Z"
      review-required: true
    provenance: discovered        # discovered by filesystem matcher
    override_applied: true        # classes modified by overrides block

  - type: tool
    id: Read
    attributes:
      description: "Read file contents"
    provenance: explicit          # declared in entities block

  - type: dir
    id: node_modules
    attributes:
      path: "node_modules/"
    provenance: projected         # declared as projection, contents not enumerated

projections:
  - type: dir
    id: node_modules
    entity_count: null            # unknown — not enumerated

fixed:
  - selector: "tool#Bash"
    constraints: { available: false }
  - selector: "network"
    constraints: { deny: "*" }
```

### 4.4 Entity provenance

Every materialized entity carries a `provenance` field:

| Provenance | Meaning |
|---|---|
| `explicit` | Declared in the `entities:` block |
| `discovered` | Found by a discovery recipe |
| `projected` | Declared as a projection (contents not enumerated) |
| `included` | Came from an included world file |

Provenance is metadata, not policy. It tells auditors *how* the entity entered the world, which informs trust decisions without encoding them.

## 5. SQL schema extensions

The world state layer adds tables to the SQL compiler's schema (see [SQL Schema Reference](../guide/sql-schema.md)):

### 5.1 New tables

```sql
-- Where each entity came from
CREATE TABLE entity_provenance (
    entity_id INTEGER PRIMARY KEY REFERENCES entities(id),
    provenance TEXT NOT NULL,        -- 'explicit', 'discovered', 'projected', 'included'
    source_file TEXT,                -- which .world.yml declared/discovered it
    matcher_name TEXT,               -- for discovered: which matcher
    override_applied BOOLEAN DEFAULT FALSE
);

-- Projections: referenced but not enumerated
CREATE TABLE projections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_id INTEGER REFERENCES entities(id),  -- the projection entity itself
    boundary TEXT NOT NULL,          -- what it covers (path glob, host pattern, etc.)
    enumerated BOOLEAN DEFAULT FALSE
);

-- World-level metadata
CREATE TABLE world_meta (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Example rows:
-- ('source', 'delegate-sandbox.world.yml')
-- ('materialized_at', '2026-04-18T14:32:00Z')
-- ('detail_level', 'full')

-- Fixed constraints (world-level, not cascade)
CREATE TABLE fixed_constraints (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    selector TEXT NOT NULL,          -- the CSS selector string
    property_name TEXT NOT NULL,
    property_value TEXT NOT NULL
);
```

### 5.2 Resolution interaction

Fixed constraints apply *after* cascade resolution. At compile time, each fixed constraint's selector is compiled to a SQL WHERE clause (using the same selector-to-SQL compiler the cascade uses) and matched entities are inserted into a materialized `fixed_constraints` table with concrete `entity_id` values — not selector text. This avoids runtime selector evaluation in SQLite.

```sql
-- fixed_constraints is materialized at compile time (entity_id, not selector)
-- The compiler runs: INSERT INTO fixed_constraints (entity_id, property_name, property_value)
--   SELECT e.id, ?, ? FROM entities e WHERE <compiled_selector>

CREATE VIEW effective_properties AS
SELECT
    rp.entity_id,
    rp.property_name,
    CASE WHEN fc.id IS NOT NULL THEN fc.property_value
         ELSE rp.property_value END AS effective_value,
    CASE WHEN fc.id IS NOT NULL THEN 'fixed' ELSE 'cascade' END AS source
FROM resolved_properties rp
LEFT JOIN fixed_constraints fc
    ON fc.entity_id = rp.entity_id AND fc.property_name = rp.property_name
```

The `fixed_constraints` table stores concrete `(entity_id, property_name, property_value)` triples. The selector text is stored separately for audit/provenance but is not evaluated at query time.

## 6. Integration with existing systems

### 6.1 CLI

```bash
# Parse world file, materialize, compile policy against it
umwelt compile policy.umw --world delegate.world.yml --target sqlite

# Materialize world state only (no policy)
umwelt materialize delegate.world.yml --level full -o snapshot.yml

# Validate world file against vocabulary
umwelt check delegate.world.yml

# Diff two materialized snapshots
umwelt diff snapshot-before.yml snapshot-after.yml
```

### 6.2 Python API

```python
from umwelt.world import load_world, materialize

# Load and materialize
world = load_world("delegate.world.yml")
snapshot = materialize(world, level="full")

# Use with SQL compiler
import sqlite3
from umwelt.compilers.sql import compile_to_db
from umwelt.compilers.sql.dialects import SQLiteDialect

con = sqlite3.connect("policy.db")
compile_to_db(con, view, SQLiteDialect(), world=world)
# Note: the `world=` parameter is a planned extension to compile_to_db.
# Today it accepts `base_dir=` for matcher-based discovery.
```

### 6.3 Relationship to existing matchers

The `discover:` recipes in world files configure the same matchers that `populate.py` currently calls via the registry. The world file is the *declarative surface* for matcher configuration — today the matchers run with hardcoded defaults against the working directory; with world files, they run with explicit configuration from the YAML.

## 7. Reserved keys

The following top-level keys are reserved for future versions:

| Key | Intent | Status |
|---|---|---|
| `vars:` | Variable substitution (`${project_root}`, `${principal_email}`) | Designed, not implemented |
| `when:` | Conditional world composition (`when: { env: production }`) | Under consideration |
| `$ref` | Intake-style fragment references (`$ref: "other.world.yml#tools"`) | Designed, available via `include:` fragments |
| `version:` | World file format version | Will be required in v2 |

## 8. What this does NOT change

- The `.umw` view format (unchanged — views remain CSS)
- The cascade resolver (unchanged — still resolves policy by specificity)
- The existing Python matcher registry (unchanged — world files configure it, don't replace it)
- The compiler protocol (unchanged — compilers still consume resolved views)
- Core umwelt's vocabulary-agnostic design (unchanged — world files use whatever taxa are registered)
- The plugin registration pattern (unchanged)

## 9. Migration path

### Phase 1: World file parser + materializer
- YAML parser for `.world.yml` files
- Materialization at three detail levels
- `umwelt materialize` CLI command
- `umwelt check` validates world files against registered vocabulary

### Phase 2: SQL integration
- `entity_provenance`, `projections`, `world_meta`, `fixed_constraints` tables
- `--world` flag for `umwelt compile`
- `effective_properties` view with fixed-constraint clamping

### Phase 3: Composition + discovery
- `include:` with fragment references
- `overrides:` and `fixed:` post-processing
- Configurable discovery recipes (replacing hardcoded matcher defaults)
- `vars:` substitution

### Phase 4: Tooling
- `umwelt diff` for materialized snapshots
- `umwelt audit` combining world snapshot + policy + resolution into a single report
- IDE support for `.world.yml` (schema validation, completion)

## 10. Design principles

1. **The world file is a capability grant.** It defines what a delegate holds — not what it's allowed to do (that's policy), but what exists in its reality. Outside the world file, nothing exists from the delegate's perspective.

2. **YAML because no conditionals.** The format enforces the separation: if you want conditional behavior, you must write it in CSS (policy). The world file is a fact sheet, not a rule engine.

3. **Projections over enumeration.** Not everything needs to be listed. A projection says "this exists, it's big, I'm not listing the contents." Policy can still match on the projection entity. This keeps world files tractable.

4. **Fixed constraints are physics, not policy.** They represent hard boundaries — what the environment *cannot provide*, regardless of what policy says. The screen is 1920px wide; no CSS changes that.

5. **Materialization is audit.** The snapshot answers "what could the delegate see?" with a concrete inventory. It's the world state at a point in time, with provenance, at whatever detail level the auditor needs.

6. **Composition over completeness.** World files layer: base + project + delegation. Each layer is independently authored, versioned, and auditable. The full world is the composition.
